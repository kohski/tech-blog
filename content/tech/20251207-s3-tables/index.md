# 1. はじめに：なぜS3 Tablesを選んだのか

製造業のデータ基盤を構築する中で、S3 Tables（Iceberg）を使って「最新版管理」を実現する方法を試してみました。
この記事では、なぜS3 Tablesを選んだのか、どのように設計・実装したのか、そして実際に使ってみた本音をまとめます。

## 1.1 製造業で扱うデータ

製造業で扱うデータは、大きく2つのタイプがあるかなと思います。

### イベントログ型

特徴
- 温湿度などのIoTデータがこっちに該当します
- 時系列で積み上がっていくデータです
- 基本的に追記のみで、過去のデータは変更されません

一般的なアプローチ
- Firehose経由でS3に保存するのがセオリーです
- 日付でパーティションを切って、どんどん積んでいく設計が一般的です

### 状態管理型

特徴
- イベントログではないデータです
- 同じIDに対して状態が変更されるデータです
- 「正」が1件に定まるデータです

一般的なアプローチ
- DynamoDBやRDBがセオリーです
- ただし、DynamoDBの場合は柔軟なクエリが難しいです
- RDBは大容量になると辛いです
  - コストもかかりますし、運用も大変です

## 1.2 追記型データレイクへの違和感

状態管理型のデータを扱う際、従来の追記型データレイクの設計には違和感がありました。

日付で積むだけの設計の限界
- 最新の状態がどこにあるのか、すぐにはわかりません
- 毎日全件を保存すると、データ量が膨大になります

重複排除前提がしんどい
- 最新版だけを取得するには、毎回重複排除の処理が必要です
- クエリが複雑になり、パフォーマンスも悪くなります

最新がぱっとわからない問題
- 「このIDの最新の状態は？」という問いに答えるのが大変です
- 分析のたびに最新版を抽出する処理が必要になります

## 1.3 S3 Tablesを使ってみた

これらの課題を解決するために、S3 Tables（Iceberg）を試してみました。

以下の要件にはまりました
- 低頻度更新：状態はそんなに頻繁に変わらない（せいぜい日次）
- 大量データ：数百万件のレコードを扱う
- 状態更新型のデータ：同じIDに対して状態が更新される

選んだ理由
- コストが安い：S3ベースなので、ストレージコストが安いです
- SQLが書ける：AthenaでSQLが書けるので、分析がしやすいです
- Upsertができる：MERGE文でUpsertができるので、最新版管理が可能です
- 後続のデータアクセスが豊富：
  - Athenaで直接クエリできます
  - Redshift経由でBedrock Knowledge Baseからの問い合わせが可能です

# 2. アーキテクチャとデータ設計

## 2.1 構成概要

実際に構築した構成は以下の通りです。オーソドックスなデータ可視化の設計です。

- S3 Tables（Iceberg）：データの保存形式
- Athena：SQLクエリエンジン
- Glue：カタログ管理とETL処理
- QuickSight：可視化ツール

## 2.2 レイヤ設計

データレイクは、用途と安定度で3つのレイヤに分けました。

IcebergはMERGEによるUpsertが可能とはいえ、  
正のデータを頻繁に書き換える前提で使うものではないと考えています。

そのため、

- 業務的に意味が安定しているデータ
- 正として扱うデータ

と、

- 可視化要件で揺れやすいもの
- 集計の試行錯誤が発生する部分

を意図的に分離しました。

### raw：受け皿
- 外部システムの生CSVをそのまま保存
- S3 Tablesは使わず、通常のS3に配置
- 日付パーティションのみ

### core：最新版テーブル（正のデータ）
- 最新版のみを保持
- S3 Tables（Iceberg）で管理
- MERGEでUpsert

### mart：可視化用
- 可視化用の集計済みテーブル
- coreから作成
- 変更前提のレイヤ

Icebergで更新はできても、  
「書き換える場所を選ぶ」ことでデータを守る設計にしています。

# 4. 使ってわかったこと（本音）

## 4.1 よかったこと

### Upsertできる安心感と簡潔さ

MERGE文でUpsertができるのは、とても安心感があります。
重複排除の処理が不要で、最新版管理が簡単にできるため、クエリがシンプルになります。

### テーブルとして信用できる

Icebergテーブルは、テーブルとして信用できます。

- ACIDトランザクションをサポートしています
- スキーマの進化が可能です
- タイムトラベルが使えます

## 4.2 つらかったこと

### MERGEは魔法ではない

MERGE文は便利ですが、魔法ではありません。

- 主キー設計が必須です。また、主キーを定めることはできますが、制約を課すものではありません。普通に重複データもINSERTすることは可能です。
- 1テーブルに対して1並列でしか処理ができない：同じテーブルに対して複数のMERGE文を同時に実行することはできません。並列処理が必要な場合は、テーブルを分割するなどの設計が必要です

### LakeFormationの権限周りが複雑

LakeFormationを使っている場合、権限周りが複雑です。

- テーブルへのアクセス権限、カラムレベルの権限、データレイクの権限など、複数の権限設定が必要です
- IAMロールとLakeFormationの権限の関係がわかりづらいです
- 権限エラーが発生したときの原因特定が難しいです
- 開発環境と本番環境で権限の設定方法が異なる場合があり、環境間の差異に注意が必要です

# 5. まとめ：S3 Tablesは「最新版管理 × 分析」の妥協点として優秀

S3 Tables（Iceberg）は、状態管理型データを扱う上で、良い妥協点だと思いました。

DBでもない
- RDBのような厳密な制約はありません
- トランザクションの保証も限定的です

S3でもない
- 単なるファイルストレージではありません
- テーブルとして扱えます
- SQLが書けます

ちょうどいい場所
- 最新版管理ができます
- 分析がしやすいです
- コストが安いです

製造業との相性は良い
- 低頻度更新の状態管理型データに適しています
- 大量データを扱えます
- SQLで分析できるので、既存のスキルが活かせます

ただし、主キー設計を正しく行うこと、適材適所で使うことが重要です。
RDBの代替として使うのではなく、分析用途として使うことが重要だと思います。
